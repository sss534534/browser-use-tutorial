
## browser-use Agent 执行过程深度分析

基于代码库的详细研究，以下是 browser-use agent 的完整执行流程：

### **1. 整体执行架构（High-Level Flow）**

```
┌─────────────────────────────────────────────────────────────┐
│                      Agent.run()                             │
│              (主执行循环，最多 max_steps 次)                  │
└────────────────┬────────────────────────────────────────────┘
                 │
        ┌────────▼────────────┐
        │ 初始化 (Init Phase)  │
        └────────┬────────────┘
                 │
        ┌────────▼──────────────────────────┐
        │ 1. 启动浏览器会话                  │
        │ 2. 初始化事件总线                  │
        │ 3. 注册技能作为动作                │
        │ 4. 执行初始动作                    │
        └────────┬──────────────────────────┘
                 │
        ┌────────▼──────────────────────────────┐
        │      主循环：while step <= max_steps   │
        │    (每次迭代调用 _execute_step)       │
        └────────┬──────────────────────────────┘
                 │
        ┌────────▼──────────────────────┐
        │  _execute_step()               │
        │  ├─ on_step_start回调          │
        │  ├─ 执行 step()                 │
        │  │  (包含3个阶段)               │
        │  └─ on_step_end回调            │
        └────────┬──────────────────────┘
                 │
        ┌────────▼─────────────────────────────┐
        │   task完成 或 超过max_steps        │
        │   └─ 关闭浏览器，停止事件总线      │
        └───────────────────────────────────────┘
```

### **2. 单个步骤详细执行流程（Step内部 3 阶段）**

```python
async def step(self, step_info: AgentStepInfo | None = None) -> None:
    """
    执行单个步骤 = 3个阶段
    """
    
    # ════════════════════════════════════════════════════════════
    # 📍 Phase 1: 准备上下文（_prepare_context）
    # ════════════════════════════════════════════════════════════
    browser_state_summary = await self._prepare_context(step_info)
    # 获取：
    #   - 当前DOM状态和截图
    #   - 可交互元素列表
    #   - 基于页面的可用操作
    #   - 辅助功能树
    
    # ════════════════════════════════════════════════════════════
    # 📍 Phase 2: 获取模型输出并执行动作（_get_next_action + _execute_actions）
    # ════════════════════════════════════════════════════════════
    await self._get_next_action(browser_state_summary)
    # 发送给 LLM：
    #   - 系统提示（system prompt）
    #   - 浏览器状态消息（browser state message）
    #   - 历史上下文（agent history items）
    #   - 当前任务描述
    #   - 截图和DOM内容
    # LLM 返回：
    #   - AgentOutput 结构化输出
    #   - 包含 thinking, memory, next_goal, action[]
    
    await self._execute_actions()
    # 根据 LLM 输出执行动作：
    #   - 点击（click）
    #   - 输入文本（type）
    #   - 滚动（scroll）
    #   - 等等（up to 5 actions per step）
    # 记录结果到 ActionResult
    
    # ════════════════════════════════════════════════════════════
    # 📍 Phase 3: 后处理（_post_process）
    # ════════════════════════════════════════════════════════════
    await self._post_process()
    # - 更新历史记录
    # - 检查任务是否完成 (is_done)
    # - 内存压缩（如果需要）
    # - 更新步骤计数器
```

### **3. LLM 交互详细流程**

#### **输入消息结构：**
```
┌─────────────────────────────────────────────────┐
│           LLM 输入消息堆栈                      │
├─────────────────────────────────────────────────┤
│ 1. 系统消息 (SystemMessage)                     │
│    - 角色定义：你是 AI 浏览器代理               │
│    - 可用工具列表（click, type, scroll...）    │
│    - 指导原则和最佳实践                        │
│                                                 │
│ 2. 状态消息 (State Message) - 建立当前上下文   │
│    - 当前 URL                                  │
│    - 当前任务                                  │
│    - 浏览器状态摘要                            │
│    - 截图 (base64 encoded)                    │
│    - DOM 元素列表（可交互）                    │
│                                                 │
│ 3. 上下文消息 (Context Messages) - 历史记录    │
│    - 之前步骤的输出和结果                      │
│    - 记忆（compressed if needed）              │
│    - 错误和恢复信息                            │
└─────────────────────────────────────────────────┘
```

#### **输出结构（AgentOutput）：**
```python
class AgentOutput(BaseModel):
    thinking: str | None          # 思考过程（使用 extended thinking 时）
    evaluation_previous_goal: str  # 评估上一步是否达成目标
    memory: str                   # 长期记忆（总结当前步骤学到的内容）
    next_goal: str                # 下一步的目标
    action: list[ActionModel]     # 要执行的动作列表（1-5个）
    current_plan_item: int | None # 当前计划项目
    plan_update: list[str] | None # 计划更新
```

### **5. 动作执行流程（_execute_actions）**

```python
动作执行步骤：
1. 解析 LLM 输出中的动作列表
2. 对每个动作：
   a. 验证动作参数
   b. 通过 Tools 服务执行
   c. 收集 ActionResult（成功/失败/错误）
   d. 等待网络空闲或指定延迟

3. 实时反馈：
   - 元素突显（highlight selected elements）
   - 截图记录
   - 错误捕获和恢复

4. 内存管理：
   - 更新 long_term_memory
   - 记录 action_results
```

### **7. 完成条件和判断逻辑**

Agent 认为任务完成的条件：

```python
if self.history.is_done():
    # 触发完成流程：
    await self.log_completion()
    
    # 可选：运行 Judge 验证
    if self.settings.use_judge:
        await self._judge_and_log()
    
    # 触发完成回调
    if self.register_done_callback:
        await self.register_done_callback(self.history)
    
    # 返回 AgentHistoryList
    return self.history
```

### **8. 超时和错误处理**

```python
每个步骤的超时保护：
step_timeout = TIMEOUT_Step (默认 180 秒)
        ↓
await asyncio.wait_for(
    self.step(step_info),
    timeout=self.settings.step_timeout
)

错误处理：
try:
    await self.step()
except TimeoutError:
    记录超时错误，继续下一步
except Exception as e:
    调用 _handle_step_error(e)
    → 可能触发重试或标记步骤失败
finally:
    调用 _finalize(browser_state_summary)
    → 更新截图、清理资源
```

### **9. 云同步和远程浏览器支持**

```python
BrowserSession 支持两种模式：

模式1：本地浏览器
├─ 启动本地 Chrome 进程
├─ 通过 CDP (Chrome DevTools Protocol) 控制
└─ 使用 cdp-use 库进行类型化 CDP 调用

模式2：云浏览器
├─ 通过 cloud_profile_id 连接
├─ 使用代理（可选国家代码）
├─ 云端超时管理
└─ CDP 仍通过 cdp-use
```

### **10. 关键文件和模块映射**

| 模块 | 文件 | 职责 |
|------|------|------|
| **Agent** | `browser_use/agent/service.py` | 主执行循环、步骤管理、LLM 交互 |
| **BrowserSession** | `browser_use/browser/session.py` | 浏览器生命周期、CDP 连接、事件总线 |
| **Tools** | `browser_use/tools/service.py` | 动作注册、执行映射 |
| **DomService** | `browser_use/dom/service.py` | DOM 提取、元素突显、可访问性树 |
| **MessageManager** | `browser_use/agent/message_manager/service.py` | 历史记录、内存管理、消息构建 |
| **Judge** | `browser_use/agent/judge.py` | 任务完成验证（可选） |

这个架构设计确保了：
- **异步高效**：全面使用 async/await
- **事件驱动**：多个服务通过事件总线协调
- **容错能力**：完整的错误处理和超时管理
- **可观测性**：详细的日志和观察钩子
- **灵活扩展**：支持自定义技能、工具和 LLM
