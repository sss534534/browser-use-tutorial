# 3. Agent 架构模式深入分析

## 🏗️ Agent 的三层架构模式

browser-use 的 Agent 采用了 **分离关注点的多层架构**，形成了一个高度模块化、可测试和可维护的系统。

```
┌────────────────────────────────────────────────────────┐
│         Agent Layer (业务流程编排)                      │
│  ├─ run() / step()                                    │
│  ├─ _prepare_context() / _get_next_action()          │
│  └─ _execute_actions() / _post_process()             │
└────────────────────┬─────────────────────────────────┘
                     │
┌────────────────────┴─────────────────────────────────┐
│    Message Management Layer (LLM 交互抽象)            │
│  ├─ MessageManager                                   │
│  ├─ SystemPrompt                                     │
│  ├─ AgentMessagePrompt                               │
│  └─ Message History (compaction + caching)           │
└────────────────────┬─────────────────────────────────┘
                     │
┌────────────────────┴─────────────────────────────────┐
│      LLM Layer (多提供商支持)                         │
│  ├─ BaseChatModel (抽象接口)                        │
│  ├─ ChatBrowserUse (专有模型)                       │
│  ├─ ChatOpenAI / ChatAnthropic / ChatGoogle         │
│  └─ ainvoke(messages, output_format)                │
└────────────────────────────────────────────────────────┘
```

## 1️⃣ Agent Layer - 业务流程层

### **核心设计：Generic + Generics**

```python
class Agent(Generic[Context, AgentStructuredOutput]):
    """支持自定义类型的通用 Agent"""
    
    # Context 类型参数（当前实现中为 BrowserSession）
    # AgentStructuredOutput 类型参数（用户定义的输出格式）
```

**为什么使用 Generics？**
- 🎯 **类型安全**：编译时检查，IDE 自动完成
- 🔄 **灵活复用**：同一 Agent 类支持不同的上下文类��
- 📊 **结构化输出**：用户可定义自己的输出格式

```python
# 示例 1: 标准 Agent（通用输出）
agent = Agent[BrowserSession, Dict[str, Any]](
    task="Search for AI news",
    llm=ChatBrowserUse(),
)

# 示例 2: 结构化输出 Agent（自定义输出模型）
class NewsResult(BaseModel):
    title: str
    url: str
    summary: str

agent = Agent[BrowserSession, NewsResult](
    task="Search for AI news",
    llm=ChatBrowserUse(),
    output_model_schema=NewsResult,
)

# 使用
history = await agent.run(max_steps=100)
result = history.structured_output  # type: NewsResult
```

## 2️⃣ Message Management Layer - 消息管理层

### **架构图：消息的完整生命周期**

```
消息生命周期：

初始化
    ↓
SystemPrompt (静态系统提示)
    ├─ 任务定义
    ├─ 可用动作列表
    ├─ 思考格式 (thinking)
    └─ 评估格式 (evaluation_previous_goal)
    │
    ↓
每步迭代
    ├─ prepare_step_state()
    │   ├─ 浏览器状态总结
    │   ├─ 前一步结果
    │   └─ 当前内存
    │
    ├─ build_messages()
    │   ├─ SystemMessage (系统提示)
    │   ├─ 历史消息 (可压缩)
    │   ├─ 当前状态消息
    │   └─ 图像 (如果 use_vision=True)
    │
    ├─ LLM 调用
    │   ├─ 发送消息到 LLM
    │   └─ 获取 AgentOutput
    │
    └─ maybe_compact_messages()
        ├─ 检查历史长度
        ├─ 如果超过 max_history_items
        │   └─ 压缩摘要 (使用 page_extraction_llm)
        └─ 更新历史

最终化
    ↓
保存完整历史 (不删除)
```

### **抽象基类：BaseChatModel**

```python
class BaseChatModel(ABC):
    """所有 LLM 实现的抽象基类"""
    
    @property
    def model(self) -> str:
        """模型名称"""
        pass
    
    @property
    def supports_vision(self) -> bool:
        """是否支持视觉"""
        pass
    
    @property
    def supports_thinking(self) -> bool:
        """是否支持扩展思考"""
        pass
    
    @abstractmethod
    async def ainvoke(
        self,
        messages: list[BaseMessage],
        output_format: type[T] | None = None,
        **kwargs: Any
    ) -> ChatInvokeCompletion[T] | ChatInvokeCompletion[str]:
        """
        异步调用 LLM
        
        Args:
            messages: 消息历史
            output_format: Pydantic 模型用于结构化输出
            **kwargs: 提供商特定的参数
        
        Returns:
            completion: 解析后的 AgentOutput
            usage: token 使用情况
        """
        pass
```

### **SchemaOptimizer：智能优化 JSON Schema**

```python
class SchemaOptimizer:
    """优化 JSON Schema 以减小 token 成本"""
    
    @staticmethod
    def create_optimized_json_schema(
        model: type[BaseModel],
        remove_min_items: bool = False,
        remove_defaults: bool = False,
    ) -> dict:
        """
        优化策略：
        1. 移除不必要的 $defs (深层定义)
        2. 移除 min_items 约束 (不影响逻辑)
        3. 移除默认值 (LLM 自动推导)
        4. 简化属性描述
        5. 合并类型定义
        
        效果：Schema 大小 ↓ 30-50%，Token 成本 ↓ 15-25%
        """
        schema = model.model_json_schema()
        
        # 移除不必要的元数据
        if remove_min_items and 'min_items' in schema:
            del schema['min_items']
        
        if remove_defaults:
            for prop in schema.get('properties', {}).values():
                if 'default' in prop:
                    del prop['default']
        
        # 简化引用
        if '$defs' in schema:
            schema['$defs'] = {
                k: v for k, v in schema['$defs'].items()
                if k in schema.get('properties', {})
            }
        
        return schema
```

async def _get_next_action(self, browser_state_summary: BrowserStateSummary) -> None:
    """获取下一个动作（LLM 交互）"""
    
    # 1️⃣ 准备本步的浏览器状态
    self._message_manager.prepare_step_state(
        browser_state_summary=browser_state_summary,
        model_output=self.state.last_model_output,  # 前一步的 LLM 输出
        result=self.state.last_result,              # 前一步的执行结果

        llm=self.llm,
        settings=self.settings.message_compaction,

    input_messages = self._message_manager.get_messages()
    
    # 4️⃣ 调用 LLM
    model_output = await self._get_model_output_with_retry(input_messages)
    
    # 5️⃣ 保存输出用于后续步骤
    self.state.last_model_output = model_output

### **Phase 3: 消息构造详细流程**

```python
def get_messages(self) -> list[BaseMessage]:
    """构造完整的消息列表"""
    
    messages: list[BaseMessage] = []
    
    # 第 1 部分：系统消息
    system_content = self._build_system_message()
    messages.append(SystemMessage(content=system_content))
    
    # 第 2 部分：初始任务 + 截图
    initial_user_message = self._build_initial_user_message()
    messages.append(UserMessage(content=initial_user_message))
    
    # 第 3 部分：历史对话（可能被压缩）
    for history_item in self.state.history[-max_history_items:]:
        # 添加 LLM 的上一步输出
        messages.append(AssistantMessage(
            content=format_agent_output(history_item.model_output),
        ))
        
        # 添加该步的执行结果
        user_feedback = self._build_step_feedback(
            model_output=history_item.model_output,
            result=history_item.result,
            browser_state=history_item.state,
        )
        messages.append(UserMessage(content=user_feedback))
    
    # 第 4 部分：当前状态消息
    current_state_message = self._build_current_state_message()
    messages.append(UserMessage(content=current_state_message))
    
    return messages
```

def _build_initial_user_message(self) -> list[ContentPart]:

            source=ContentPartImageSource(

## 🔄 消息压缩机制（Memory Compaction）

### **何时触发压缩**

```python
async def maybe_compact_messages(self, llm, settings):
    """条件性地压缩历史"""
    
    # 检查触发条件
    if len(self.state.history) <= settings.max_history_items:
        return False  # 不需要压缩
    
    # 计算压缩前的 token 数
    messages_before = self.get_messages()
    tokens_before = llm.count_tokens(messages_before)
    
    if tokens_before < settings.compaction_threshold:
        return False  # Token 数量还可以
    
    # 触发压缩
    await self._compress_history(llm)
    return True
```

### **压缩算法**

```python
async def _compress_history(self, llm: BaseChatModel) -> None:
    """使用 LLM 压缩历史"""
    
    # 1️⃣ 选择要压缩的历史（通常是最早的 20 步）
    history_to_compress = self.state.history[:20]
    
    # 2️⃣ 构建压缩提示
    compression_prompt = f"""
    Please summarize the following steps into a bullet-point memory.
    Include only the most important findings:
    
    {format_history(history_to_compress)}
    
    Summary (bullet points, be concise):
    """
    
    # 3️⃣ 调用 page_extraction_llm 生成摘要
    messages = [SystemMessage(content=compression_prompt)]
    summary = await self.page_extraction_llm.ainvoke(messages)
    
    # 4️⃣ 更新状态
    self.state.compacted_memory += f"\n{summary.completion}"
    self.state.history = self.state.history[20:]  # 删除已压缩的
    self.state.compaction_count += 1
    
    # 5️⃣ 计算节省的 token
    messages_after = self.get_messages()
    tokens_saved = count_tokens(messages_before) - count_tokens(messages_after)
    self.logger.info(f"Compacted history. Saved {tokens_saved} tokens.")
```

### **压缩效果计算**

```
示例：100 步任务

无压缩：
步骤 1:   历史 [步骤 1]
步骤 2:   历史 [步骤 1-2]
步骤 3:   历史 [步骤 1-3]
...
步骤 100: 历史 [步骤 1-100]

总 token = 1 + 2 + 3 + ... + 100 = 5050

有压缩（每 20 步压缩一次）：
步骤 1-20:   完整历史 (1+2+...+20 = 210 token)
步骤 21-40:  摘要 + 最近 5 步 (25 + 25*5 = 150 token)
步骤 41-60:  摘要 + 最近 5 步 (150 token)
步骤 61-80:  摘要 + 最近 5 步 (150 token)
步骤 81-100: 摘要 + 最近 5 步 (150 token)

总 token ≈ 210 + 150*4 = 810

节省：5050 / 810 ≈ 6.2x
```

| 设计 | 优点 | 例子 |

| **多层分离** | 每层职责清晰 | Agent 不需要知道 LLM 如何实现 |
| **Generic 类型** | 类型安全 + 灵活 | 支持自定义 output_model_schema |
| **MessageManager** | 消息管理集中 | 压缩、缓存、敏感数据处理 |
| **BaseChatModel** | 提供商无关 | 轻松切换 LLM |
| **SchemaOptimizer** | 成本优化 | Token 消耗 ↓30-50% |
| **消息压缩** | 上下文管理 | 支持 100+ 步任务 |
| **URL 缩短** | 消息优化 | 长 URL 自动缩短 |
| **重试机制** | 容错性 | LLM 失败自动重试 |

## 总结：Agent 架构的三层精妙设计

```
┌─ Application Layer ───────────────────┐
│ 简单直观的 API                        │
│ agent = Agent(...)                   │
│ await agent.run()                    │
└──────────────────────────────────────┘

┌─ Message Management Layer ────────────┐
│ 复杂的消息构造逻辑被隐藏              │
│ • 历史管理                            │
│ • 压缩和优化                          │
│ • 敏感数据处理                        │
└──────────────────────────────────────┘

┌─ LLM Layer ────────────────────────────┐
│ 多提供商支持，统一接口                │
│ • OpenAI / Anthropic / Google / ...  │
│ • 结构化输出 + Schema 优化            │
└──────────────────────────────────────┘
```

这样的设计使得：
✅ **用户**：简单易用的 API
✅ **开发者**：清晰的模块分离，易于维护
✅ **系统**：高效的消息处理和 LLM 调用
✅ **成本**：通过优化减少 token 消耗 30-50%
